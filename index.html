WHY THIS PROJECT

왜 만들었나, 목적과 방향

혼자 사는 사람 + 맞벌이 부부가 늘면서 "오늘 산책을 못 시켜준다", "긴급한 일정에 반려동물을 누구에게 맡기지" 같은 일상의 구체적인 어려움이 커지고 있습니다. 동시에 잃어버린 반려동물·건강 관리·이상 행동 같은 영역에는 흩어진 채널만 있을 뿐 한 곳에서 해결할 통합 서비스가 부족합니다. 이 프로젝트는 그 사이를 메우는 것이 목적입니다.

우리가 본 문제problem
  • 1
    산책·픽업 매칭 시장이 흩어져 있다. 동네 단톡방, 당근, 별도 앱이 각자 — 신뢰·후기·요금이 일관되지 않음.
  • 2
    잃어버린 반려동물 신고가 즉시 확산되지 않는다. 게시판·SNS 방식이라 위치 기반 푸시가 어렵고, 제보 응답률이 낮음.
  • 3
    건강·간식·산책 데이터가 분리되어 있다. 보호자가 직접 적어두지 않으면 동물병원·시터에게 컨텍스트 전달 어려움.
  • 4
    관리자/운영 도구가 없는 UGC 서비스는 빠르게 노이즈로 무너진다. 신고·차단·모더레이션이 처음부터 필요.
우리의 방향solution
  • 1
    한 앱에서 산책·픽업·매칭·건강·구조를 모두. 양면 시장(보호자 ↔ 워커)을 단일 백엔드로 통합.
  • 2
    위치 기반 푸시 + 모더레이션. 잃어버린 반려동물 신고가 인근 사용자에게 즉시 도달, 제보는 관리자 모더레이션 후 보호자에게 노출.
  • 3
    AI 간식 분석·LLM 챗. 사진 한 장으로 영양·위험 평가, 모르는 부분은 LLM 으로 답변. 5개 프로바이더 추상화로 비용·가용성 분산.
  • 4
    운영 친화 — 모더레이션·차단·신고·통계를 첫날부터. Apple Guideline 1.2 UGC 요건을 코드 차원에서 충족.

DESIGN · HISTORY

UPDATE 대신 *회전 (rotation)*

등급 변경 시 기존 행을 update 하지 않고 EXPIRED 로 보존 + 새 ACTIVE INSERT. 언제 어떤 plan 이었는지 추적 가능 — 결제 환불·분쟁·통계 모두 유리합니다.

단점은 행 누적이지만 active_marker generated column (V91) 으로 인덱스만 차지하고 디스크 비용 0. *현재 행 보호 + 이력 누적* 을 동시에 달성한 패턴.

RUNTIME · LAZY

Cron 없이 *조회 시점 만료*

getActivePlan(userId) 호출 시 expired_at 이 지났으면 그 자리에서 status=EXPIRED 로 갱신하고 FREE 반환. 별도 배치 작업 없음 — 운영 부담 감소.

약점은 "조회 안 한 사용자는 DB 상 ACTIVE 그대로" — 통계에 약간 오차. 다만 어드민 stats 는 *count* 쿼리라 실제 사용량과 큰 차이 없고, 운영 사용자 100명 미만에선 cron 도입이 오버엔지니어링.

CONFIG · DECOUPLE

한도는 *DB 에 분리* (app_settings)

플랜별 한도 (FREE=1, BASIC=3, PREMIUM=10) 를 코드 상수가 아닌 app_settings 테이블에 둠. 한도 변경 = DB 한 행 UPDATE. 코드 / 배포 불필요.

키 명명 규칙: subscription.{feature}.{cycle}_limit.{plan}. 새 feature 추가 시 settings 행만 추가하면 코드 영향 최소. 운영에서 *플랜 정책 실험* 이 쉬워짐.

DESIGN · GROUP

별도 테이블 없이 *self-reference* 그룹 산책

한 번에 여러 펫을 산책시키는 보호자가 흔합니다. 그룹마다 별도 walks_group 마스터 테이블을 두는 대신, 대표 walk 가 자기 id 를 group_walk_id 에 박는 self-reference 패턴 (V87 마이그레이션) 으로 해결했습니다.

ROUTE · NORMALIZE

경로는 *리더에게만*, 호출은 *어디서든*

GPS route_points 가 그룹 멤버 수만큼 곱해 저장되면 디스크와 polyline 계산이 N배. 리더 walk_id 에만 한 벌 저장하고 멤버 walk 는 경로 없음 — *멤버 id 로 호출해도 서버가 대표 id 로 재할당* 해 클라이언트는 그룹 여부를 몰라도 됩니다.

완료 시점에 한 번 polyline 인코딩 → preview_polyline (TEXT) 컬럼에 캐시. 카드 미리보기는 route_points 를 다시 집계하지 않아 조회당 DB hit 1회 로 충분합니다.

BROADCAST · CAP

*반경 + 동의 + 5000명 cap* 으로 막은 알림 폭주

신고 발송 대상은 단순한 "전체" 가 아닙니다. lostpetNotificationConsent = Y 동의 사용자 + 신고 좌표로부터 0.5~5km Haversine 반경 + 신고자 본인 제외 + 거리순 정렬 후 LIMIT 5,000 — 네 조건 모두 한 QueryDsl 쿼리에서 처리합니다.

cap 이 없으면 한 신고가 도시 전체에 푸시를 쏴 FCM timeout 과 *스팸성 알림* 양쪽이 터집니다. cap + 거리순 ORDER BY 로 가장 가까운 5000명 만 — 사용자 경험과 운영 안정성을 동시에 잡았습니다.

SNAPSHOT · IMMUTABLE

신고 사진은 *그 시점 그대로* 박는다

신고 등록 시 첨부 사진이 없으면 그 시점의 펫 프로필 사진을 lost_pet_photos 에 스냅샷으로 INSERT. 이후 보호자가 펫 프로필을 바꿔도 신고에 보이는 사진은 그대로 — 잃어버린 *그 날* 의 외모를 인근 사용자가 봐야 하기 때문입니다.

기존 데이터는 V86 마이그레이션으로 backfill. 조회 로직은 단일 출처 (lost_pet_photos) 만 보면 되어 단순해지고, 펫 도메인을 런타임에 조회하던 N+1 도 사라졌습니다.

LIFECYCLE · HYGIENE

30일 자동 종료 + 끌어올림 쿨다운

신고는 SEARCHING → FOUND/CLOSED 단방향 상태 머신입니다. bumped_at + 30일 지나면 새벽 3시 스케줄러가 자동 CLOSED — 운영자가 일일이 정리할 필요 없이 데이터 위생이 유지됩니다.

ARCHITECTURE

추상화는 비용이 아니라, 미래의 선택지

도메인이 하나뿐일 땐 직접 의존이 가장 빠른 길처럼 보입니다. 그런데 한 도메인이 다른 도메인을 import 하는 순간, 그 둘은 앞으로 함께만 옮길 수 있게 됩니다. 단기 비용을 아끼다 장기 선택지를 잃는 거죠.

그래서 Port 어댑터 패턴을 처음부터 강제했습니다. abuse 모듈이 lostpet 의 sighting 을 직접 disable 하지 않고, admin 이 별도 API 로 모더레이션을 호출하는 식. "미루지 않고 미리 깔되, 추상화의 단위는 도메인 경계까지만" — 이게 한 명이 풀스택을 다룰 때 가장 중요한 절제였습니다.

DESIGN OPS

디자인 토큰은 한 번 정의되지 않는다

처음엔 Figma 가 단일 진실 출처(SSOT)라고 믿었어요. 그런데 막상 Flutter 로 옮기는 순간 코드가 디자인을 깎아먹기 시작합니다 — 미묘하게 어긋난 padding, "이 색이 #5BA070 인가 #5DAA72 인가" 같은 흔들림.

그래서 CSS variable 에 정의된 토큰을 Flutter AppColors 가 그대로 미러링하도록 만들었습니다. 60+ HTML 시안과 앱 코드가 같은 팔레트를 공유 — 토큰을 바꾸면 양쪽이 같이 움직입니다. 시안은 합의된 도면이고 구현은 그걸 따라가는 것 — 이 순서가 거꾸로 가면 톤이 먼저 무너집니다.

OPS

오버엔지니어링과 미래 대비의 경계

"나중에 Redis 가 필요할 거니까 지금 깔자"는 유혹이 컸어요. 다중 인스턴스, 분산 캐시, pub/sub 모두 의미는 분명하지만 — 지금 도입하면 운영 부담만 늘고 ROI 가 안 나옵니다.

타협점: 코드는 안 짜되, 인터페이스는 열어둔다. TokenBlacklistService 는 in-memory 로 시작했지만 인터페이스가 분리돼 있어 추후 Redis 로 swap 가능. refresh_tokens 는 처음부터 DB 영속으로 — Redis 였다면 재시작마다 전 사용자 강제 로그아웃이라는 사고가 났겠죠. 영속성 보장 등급을 컴포넌트별로 일찍 정해두는 게 핵심이었습니다.

TRADE-OFFS

"안 만든 것"의 목록도 결정입니다

의도적으로 미룬 항목 — 실시간 채팅(WebSocket), 멀티 인스턴스, Sentry/APM, 자동 스케일링, refresh rotation 의 다중 디바이스 race 처리. 각 항목엔 "지금 도입하면 따라오는 것들"이 명확합니다. Sentry 는 도구 도입은 쉽지만 알람 정책·플레이북이 없으면 노이즈만 되고, WebSocket 은 nginx · 연결 상태 관리 · 클러스터 동기화가 함께 옵니다.

원칙은 단순합니다 — 예측 기반이 아니라 관측 기반. 사용자 100명 미만에선 docker logs 와 grep 으로 충분합니다. 그 이상이 되면 그때 도입. 시니어 일의 절반은 "무엇을 안 만들지 정하는 일" 이라는 말, 이번에 체감했습니다.

SECURITY · RUNTIME

보안은 *광범위* 보다 *정확* 해야 한다

refresh_tokens 에 걸어둔 uk_refresh_token_hash UNIQUE 는 탈취 토큰 재사용 탐지를 위한 핵심 보안 제약이었습니다. 같은 hash 가 두 번 시도되면 그 사용자의 모든 토큰을 폐기 — 이론적으로 깔끔했죠. 그런데 운영에서 트래픽이 늘면서 정상 사용자도 같은 메커니즘에 걸리기 시작했습니다.

원인은 단순했습니다 — JWT iat/exp 가 초 단위라 같은 사용자가 같은 초에 두 번 갱신을 요청하면 payload 가 완전히 동일해서 hash 도 같아집니다. DB 입장에서 *탈취 재사용* 과 *정상 동시 발급* 이 구분되지 않았던 거예요. 트래픽 적을 땐 안 보이다가 운영에서 표면화된 전형적인 false positive.

교훈은 보안 제약 추가 시 정상 케이스가 그 제약에 걸릴 시나리오를 함께 검증 해야 한다는 것. 그리고 한 가지 보호만으로 모든 케이스를 차단하려 들면 *광범위한 차단* 이 되기 쉽고, 결국 그 보호를 우회하는 fix 가 생기면서 보안 자체가 약해집니다. JWT 에 jti(UUID) 한 줄 추가로 *정상은 매번 다른 hash, 탈취는 같은 hash* 가 되어 — 탐지는 유지하면서 false positive 만 제거. 클라이언트 가드 + jti + DB UNIQUE 의 3층 방어로 재구성했습니다. 각 층이 다른 메커니즘으로 같은 목표를 달성 — 이게 *defense in depth* 의 실전 적용이었습니다.

BEHIND THE CODE

구조가 무너지지 않게 한 코드들, behind the code

기능 단위로 보면 평범한 CRUD 도, 그 안에는 한 명이 풀스택을 다룰 때 도움이 됐던 몇 가지 장치가 있었습니다. 화면에서 보이지 않지만 구조를 떠받치는 코드 3개를 꺼내봤어요.

SESSION PROPAGATION

요청 컨텍스트가 ThreadLocal 벽을 넘게 만들기

request → async → outbound

SecurityContextHolder 는 ThreadLocal 기반이라 @Async 스레드나 Feign 클라이언트 호출에 인증 정보가 전파되지 않습니다. 매번 메서드 파라미터로 userId 를 넘기면 도메인이 인증 관심사로 오염되죠. RequestAttributes 기반 GlobalSession + Task Decorator 조합으로 같은 호출 트리 안에선 어디서든 GlobalSession.getId() 가 작동하게 만들었습니다.

common/model/GlobalSession.javapublic static Long getId() {
  // 1) 일반 요청 — RequestAttributes 에서 꺼냄
  if (RequestContextHolder.getRequestAttributes() == null) {
    // 2) @Async 스레드 — RequestAttributes 가 없을 때만 AsyncSession fallback
    return AsyncSession.isEmpty() ? null : (Long) AsyncSession.getAttribute(ID);
  }
  return (Long) RequestContextHolder.getRequestAttributes()
      .getAttribute(ID, RequestAttributes.SCOPE_REQUEST);
}

// common/component/ClonedTaskDecorator.java
public class ClonedTaskDecorator implements TaskDecorator {
  @Override
  public Runnable decorate(Runnable runnable) {
    Session session = GlobalSession.getSession();   // 부모 스레드 컨텍스트 캡처
    return () -> {
      try {
        AsyncSession.setSession(session);             // 자식 스레드에 주입
        runnable.run();
      } finally {
        AsyncSession.clear();                         // 누수 방지
      }
    };
  }
}
동일한 GlobalSession.getId() 를 컨트롤러·@Async·FeignClient 어디서 호출해도 같은 사용자 컨텍스트가 보입니다. MSA 분리 시 인바운드 필터만 추가하면 같은 모델이 그대로 적용됩니다.
SECURITY

Refresh Token 회전 + 탈취 탐지

rotation with reuse detection

Refresh token 을 그냥 검증만 하면 한 번 탈취된 토큰이 계속 유효합니다. 매 refresh 마다 X-Device-Id 헤더 기반으로 새 토큰을 발급하고 기존 토큰은 replaced_by 체인으로 무효화 — 이미 회전된 토큰을 또 사용하면 그 사용자의 모든 토큰을 일괄 폐기합니다.

auth/service/RefreshTokenService.java@Transactional
public void rotate(String oldRawToken, String newRawToken, long newExpirationMillis) {
  RefreshToken found = repository.findByTokenHash(hash(oldRawToken))
      .orElseThrow(() -> BusinessException.of(ExceptionType.REFRESH_TOKEN_INVALID));

  // 탈취 탐지: 이미 회전된 토큰을 또 사용 → 해당 user 의 모든 토큰 폐기
  if (!found.isActive()) {
    log.warn("[RefreshToken] reuse detected — revoking all tokens for userId={}",
             found.getUserId());
    repository.revokeAllByUserId(found.getUserId());
    throw BusinessException.of(ExceptionType.REFRESH_TOKEN_INVALID);
  }
  if (found.isExpired()) {
    found.revoke();
    throw BusinessException.of(ExceptionType.REFRESH_TOKEN_EXPIRED);
  }

  RefreshToken next = repository.save(RefreshToken.create(
      found.getUserId(), found.getDeviceId(), hash(newRawToken), ...));
  found.rotate(next.getId());   // active=N, replaced_by=next.id 체인 기록
}
디바이스별 격리(X-Device-Id) + 회전 체인이라 "한 디바이스 해킹 = 즉시 그 사용자 전체 무효화" 가 가능합니다. 평소엔 조용히 회전하지만 비정상 패턴을 만나는 순간 트립와이어가 됩니다.

보안 제약이 정상 사용자를 차단할 때 — false positive 인시던트

refresh hash 충돌 · V81 → jti 후속

사용자가 "아침에 로그인이 자주 풀린다" 보고. 운영 로그를 보니 POST /auth/refresh 500 + Duplicate entry for uk_refresh_token_hash 가 반복. 처음엔 단순 race condition 같았지만 한 단계 더 들어가 보니 V81 (refresh_tokens 테이블) 의 보안 제약이 정상 동시 발급을 탈취로 오인하고 있었습니다. DB DESIGN

PostgreSQL partial index 를 MySQL 에서 (V91)

generated column + UNIQUE on nullable

user_subscriptions 의 UNIQUE (user_id, status) 가 "활성 구독은 사용자당 1개" 의도였지만 EXPIRED 까지 포함하는 바람에 같은 사용자가 두 번째로 등급을 바꾸면 500 에러가 났습니다. PostgreSQL 이라면 UNIQUE ... WHERE status='ACTIVE' 한 줄로 끝이지만 MySQL 은 partial index 미지원. 의미상 원래 의도와 정확히 일치하는 제약을 어떻게 표현할지가 과제였습니다.

V91__user_subscriptions_active_only_unique.sql-- ACTIVE 일 때만 user_id, 나머지(EXPIRED/CANCELLED)는 NULL.
-- MySQL UNIQUE 는 NULL 중복을 허용 → partial unique 와 동일 효과.
ALTER TABLE user_subscriptions
    ADD COLUMN active_marker BIGINT
        GENERATED ALWAYS AS (CASE WHEN status = 'ACTIVE' THEN user_id END) VIRTUAL
        COMMENT 'ACTIVE 행에만 user_id 채워지는 가상 컬럼';

ALTER TABLE user_subscriptions DROP INDEX uk_us_user_active;

ALTER TABLE user_subscriptions
    ADD UNIQUE KEY uk_us_user_active (active_marker)
        COMMENT 'ACTIVE 구독은 사용자당 1건. NULL (EXPIRED/CANCELLED) 은 중복 허용';
VIRTUAL 컬럼이라 디스크 비용 0 (인덱스에만 저장). status 변경 시 active_marker 가 DB 레벨에서 자동 갱신되어 동시성 안전. 자바 도메인 코드는 변경 없음 — 의미상 정합. DBMS 한계를 우회하면서도 *원래 의도한 제약* 을 정확히 표현한 사례.
